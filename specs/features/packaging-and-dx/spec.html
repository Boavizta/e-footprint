Packaging & developer experience

Status: Spec — under review · 2026-06-30

Make e-footprint's supported Python versions explicit and tested, and give users who can't (or won't) match them a one-command escape hatch. No dependency or serialization changes — a benchmark settled that zstandard stays (fast, ≈0 MB to import); the install story is fixed by policy + Docker, not by changing deps.

1. Problem

An external user (EcoScan, 2026-05-19) couldn't install e-footprint on a brand-new Python (niche native deps had no wheels yet, so source builds failed). Underneath that: the library states no supported-version policy, has no CI guaranteeing what actually works, and offers no toolchain-independent way to run a model. Audience: anyone installing/running efootprint (product teams, consultants, CI users) and the maintainers.

2. Success criteria

  • The README and tech_stack.md state the supported Python versions and the policy: rolling window, floor >=3.12, no upper cap, "supported" = the latest minors all compiled deps ship wheels for (today 3.12 + 3.13), with the next minor added when its wheels land.
  • CI runs the full test suite on every push across the supported versions (3.12 + 3.13) and is green.
  • A published Docker image runs a model script in one command, independent of the local Python/toolchain.
  • On a supported Python, pip install efootprint works; on an unsupported/newer one the user has a documented path (Docker, or wait for wheels) instead of a confusing build error.
  • No change to dependencies, serialization, or JSON round-trip.

3. Scope

In: (1) support-window policy + docs; (2) migrate pyproject.toml to the PEP 621 [project] layout with explicit, curated classifiers; (3) a CI matrix (3.12 + 3.13); (4) a published Docker image (minimal runner, Docker Hub, on release); (5) bump the interface's stale CI Python pin (cross-repo).

Out: a hard Python upper cap (rejected — rolling window); any dependency / codec change (zstandard kept per benchmark); making heavy deps optional — viz (matplotlib/plotly/pyvis) and pandas are parked on the memory-optimization track (see specs/roadmap.md); the Boavizta import-time fix; multicriteria.

5. Constraints

  • Constitution §4: floor stays >=3.12; Pint/NumPy/Pandas not replaced. (Making deps optional is parked, and the pandas case explicitly needs a §4 note when taken.)
  • Constitution §1.3 (leanness): the install problem is solved with policy + CI + Docker, not by adding fallback code paths to the serialization hot path.
  • CI must honour the existing quality gates (pytest with MPLBACKEND=Agg); adding mkdocs build --strict is deferred to the SSOT-metadata roadmap item.
  • This subsumes and extends the roadmap's "Library CI workflow (planned)" item (single version → matrix).

6. Resolved decisions (2026-06-30)

  • Docker: minimal runner — python:3.12-slim, efootprint installed from PyPI, entrypoint python to run a mounted model script.
  • Registry & trigger: Docker Hub boavizta/efootprint, built and pushed on each release tag. Amended (2026-07-01): parked as premature — the Dockerfile shipped (local docker build/docker run), but the publish workflow and Docker Hub distribution did not. Revisit in October 2026; see AGENTS.md.
  • Interface CI: fold in — bump the interface CI from 3.10.12 to a 3.12 + 3.13 matrix (cross-repo task).
  • Classifiers & layout: migrate to the PEP 621 [project] layout and declare curated classifiers (:: 3.12, :: 3.13) explicitly. Verified (2026-06-30): [project] + explicit classifiers makes poetry-core emit exactly those; left to auto-generation (legacy layout, or classifiers omitted) it also advertises the untested :: 3.14 — now in poetry-core's known set and allowed by >=3.12,<4.0. Explicit classifiers keep them honest (= the tested set).
