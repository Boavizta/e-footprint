Pull-based computation — Implementation plan

Status: Plan — DRAFT from design-rich kickoff, to refine during spec-plan · 2026-06-12 · Spec: spec.html · Decision rationales inherited from the kickoff discussion (7-point decision log).

1. Approach

Replace the eager push engine (calculated_attributes lists, update_* functions, mod_objs_computation_chain / attr_updates_chain ordering, wipe-all attribution flush) with a single reactive pull engine: node = slot (object, attribute[, dict key]), computed on read, cached, and invalidated by deletion along recorded edges. Two edge kinds feed one invalidation graph: calculus edges (recorded by ExplainableObject arithmetic exactly as today — unchanged, still the explainability backbone) and structural edges ("who read this membership"), recorded by relationship read-hooks against a contextvar compute stack. Membership writes bump reverse-collection slots at the container bookkeeping's field transition — the one dependency defined by writing rather than discovered by reading.

3. Cross-cutting concerns

  • Tests: the parity property test is the central gate — random mutation sequences (input edits, relinks, list changes, add/delete) vs a from-scratch rebuild, with a recompute-counter variant asserting granularity (one-UP edits touch only expected slots) and a variant starting from a freshly deserialized minimal file (cached footprints below valueless intermediates). Benchmarks: full build, single edit + pull, load-to-ready, live 1,000-iteration loop, peak RSS, descriptor read micro-bench. Existing expected-value integration tests pass unchanged.
  • Migrations: JSON schema version bump + handler (inputs honored, stale computed values dropped); both performance fixtures regenerated in stage 1 (currently drifted: stale property keys, missing newer attributes); interface session stores migrate trivially (old sessions reload as inputs-only and recompute on pull).
  • Docs: architecture.md dependency-graph and calculated-attribute sections rewritten at archive time (reactive core pattern, slot model, serialization contract); conventions.md drops the update_* naming rules; mkdocs doc-as-code generation re-pointed from update_<attr> docstrings to getter docstrings (with the §1.4 amendment); workspace AGENTS.md note about library-driven cross-repo features already added.

4. Risks

  • Untracked reads (getter branching on non-Explainable, non-relationship state) — same blind spot as today's fine-grained path, expected clean; mitigated by a stage-2 audit of getter bodies plus the parity test catching violations behaviorally.
  • Per-pull overhead — paid per dependency edge per recompute, not per array element; budget < 5% of recompute wall time, enforced by micro-benchmark.
  • Recursion depth — pull order is the call stack; depth = longest chain (tens–low hundreds of nodes); asserted on the big fixture; iterative work-stack fallback documented if ever needed.
  • Partial-reload invalidation — cached footprints below valueless intermediates is the novel state; waves traverse serialized topology (which must include structural edges); dedicated parity variant.
  • Per-edit recompute cost under the minimal contract (every interface edit request is cold; the cone recomputes from inputs but is contained at object boundaries by neighbours' cached footprints) — benchmark on the regenerated fixture; knobs: per-slot serialize flag, future worker-memory models.
  • Stage-2 codemod errors across ~250 functions — mechanical transform + behavior-identical gate before any semantic change lands.
  • Thread/async safety — compute stack is a contextvars.ContextVar with context-manager push/pop; one System per execution context documented, as today.

5. Alternatives considered

  • Keep modeling_objects_whose_attributes_depend_directly_on_me for structural changes only — rejected: retains the hand-maintained per-class boilerplate and over-invalidates; relationship slots cover membership with one-slot precision using existing wrapper interception points.
  • Full read tracking (signals purism) — rejected: the calculus graph already records exact value-level dependencies for free; instrumenting all reads discards a working mechanism.
  • Stale values + dirty flags instead of deletion — rejected: retained values only enable equality cutoffs (irrelevant under eager wave-marking) at megabytes of RAM per slot; deletion matches today's flush semantics and frees memory.
  • Coarse single-node dicts — rejected: diverges from the entry-level ancestry explainability requires (two graph views) and regresses live-model loops ~×N(keys); granularity is engine policy behind the same API, so this remains revisable.
  • Serialize hub intermediates from day one — deferred knob; Redis RAM and JSON round-trips are the binding constraints, recompute is per-edit and contained.
  • Copy-on-write scenario overlays — deferred: would make every slot read scenario-aware; fork-the-inputs covers anticipated needs.
  • Big-bang migration — rejected: parity failures would be unattributable across mechanical and semantic changes; the pull-ALL transitional eager set makes the engine swap observationally invisible instead.

6. Constitutional notes

  • §1.1 (layer separation) — respected; the one documented back-edge (the function-local System import in the computation-chain optimizer) is deleted with the optimizer. architecture.md's note follows at archive time.
  • §1.4 (doc-as-code SSOT) — names update_<attr> method docstrings as authoritative. Bodies and docstrings survive as getters, but the rule's wording and the mkdocs/interface readers must follow. Amendment required — own commit via update-constitution, scheduled with stage 2.
  • §2.3 (JSON round-trip) — preserved under the new contract: inputs and flagged slots round-trip; absent values recompute on pull (load is staleness-resilient by construction, unlike today's detection flag).
  • §2.4 (schema migration) — version bump + handler in version_upgrade_handlers.py at stage 4.
  • §2.5 (class registration gate) — references CANONICAL_COMPUTATION_ORDER, which pull semantics likely retires; verify the instantiation-order residual use and amend the gate wording in the same constitutional commit as §1.4 if confirmed.
