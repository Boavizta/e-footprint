Pull-based computation

Status: Spec — under review · 2026-06-12

1. Problem and audience

  • Computation is eager. Every input or relationship change immediately recomputes all downstream values, and loading a saved model requires either recomputing everything or having stored every computed value. Programmatic users making several changes pay a full recomputation per change; future calibration loops (fitting model inputs to monitoring data) would be throttled by recomputing the whole model per iteration when only one target value is needed.
  • Serialized models carry every intermediate value. Measured on the large performance fixture: 28.2 MB, of which only ~1% are inputs. The interface stores this per session (twice, in two variants) and re-reads it on every request — session-store RAM and JSON round-trip latency are the binding cost, paid even when nothing is computed.
  • Impact-attribution caching is all-or-nothing. Attribution results are cached lazily but every model update wipes every cache in the system, regardless of what changed.
  • Two caching paradigms coexist (eager computed attributes vs lazily cached attribution), doubling the concepts every contributor and agent must hold.

Audience: interface end-users (request latency, infrastructure cost), library users running scripted studies or optimization loops, and the maintainer (one paradigm to evolve instead of two). Why now: this unlocks the near-final performance vision — calibration against monitoring data is on the roadmap and needs tight incremental loops, and session-store costs scale with interface adoption.

2. Success criteria

Each criterion is testable; performance criteria run against the regenerated large fixture.

  • Behavior parity: all existing expected-value integration tests pass unchanged (within float tolerance).
  • Randomized parity: a property test applies arbitrary mutation sequences (input edits, relinks, list changes, object add/delete) and compares incremental results to a from-scratch rebuild — including sequences starting from a freshly loaded minimal-format file.
  • Serialized weight: a saved system weighs ≤ 30% of today's full-snapshot format (≈ 7.8 vs 28.2 MB estimated on the large fixture), stored once instead of twice.
  • Instant load: loading a saved model performs no computation; view-request model loading is ≥ 3× faster than today.
  • Edit latency: an interface edit request (load, update, save) stays within ± 20% of today's latency.
  • Live-model loop: 1,000 iterations of “nudge one input, read one target value” on an in-memory model run ≥ 10× faster than 1,000 updates today.
  • Engine overhead: graph bookkeeping < 5% of recompute wall time; peak memory ≤ today's eager build.
  • Lazy auditability: after loading a saved model, any intermediate value's number and formula can be displayed on demand without recomputing the whole model.
  • Sankey responsiveness: every Sankey combination (column visibility, life-cycle phase, exclusions) renders from stored data with no model recomputation.
  • Update-time safety preserved: invalid edits (e.g. exceeding fixed server capacity) are still rejected at update time and the model rolls back.

3. Scope

Capability-level only. File paths, class/method/parameter names, refactor instructions, internal hypotheses, and "mirror pattern X" code-level decisions belong in plan.html, not here. If you can't say it without naming a file, class, or method, it's plan content.
In scope
  • Pull-based computation: values are computed when read, cached, and invalidated precisely through the dependency graph — including relationship changes (relinking objects, adding/removing objects or list members).
  • One caching paradigm for everything, impact attribution included: invalidation follows the graph at every level; no wholesale cache wipes.
  • Transactional updates preserved: computation errors surface at update time with rollback; the set of eagerly refreshed outputs is configurable (system footprints by default, nothing for tight loops).
  • Minimal persistence contract: inputs, each impact source's two footprint timeseries (usage and fabrication), the system total, a condensed impact-repartition summary (flows summed over time, filled on first Sankey use), and the values-free calculation graph.
  • Interface adaptation: load/save under the new contract, results pages, Sankey served from the stored summary, audit drill-down computing lazily.
  • Removal of the unused simulation machinery, with rebuild recipes documented for when what-if / calibration features land.
  • Staged delivery: library-only releases first, one coordinated cross-repo step last, each stage gated by passing tests and parity.
Out of scope (this iteration)
  • Rebuilding simulation features (what-if API, forked-model sandbox, dated scenarios) — recipes recorded in the plan.
  • Copy-on-write scenario overlays inside the engine.
  • Keeping live models in worker memory across interface requests (future optimization, noted as a knob).
  • The calibration feature itself.
  • Reading computed values from old-format files: inputs are honored, stale computed values are ignored and recomputed.

5. Constraints

  • Parity is non-negotiable: identical results (float tolerance) for every existing model.
  • Update-time error surfacing is part of the interface contract and must be preserved.
  • Binding performance constraints are session-store RAM and JSON round-trip latency (paid on every request); CPU recompute is paid only on edits. Design trade-offs favor smaller persisted state over avoided recomputation.
  • No new runtime dependencies — the engine uses the standard library only.
  • One model, one execution context (no concurrent mutation of a system), as today.
  • Constitution: layer separation (§1.1) is respected — and a documented back-edge is removed outright. One constitutional wording amendment is required around doc-as-code naming (§1.4) plus a quality-gate wording review (§2.5); both handled as separate deliberate commits (see plan.html §6). JSON schema version bumps with a migration handler (§2.4).
  • Cross-repo sequencing: the first three delivery stages are library-only releases invisible to the interface; only the final stage is coordinated across repos.
