Attribution-logic revamp — Implementation plan

Status: Plan v2 (atom model) — under review · 2026-06-10 · Spec: analysis.md + edge-analysis.md (amended 2026-06-10: relay-weight kinds, → EdgeFunction flows, service-base term, naming clarifications). Scope: end-game structure only; migration steps deferred. No backward compatibility.

Settled decisions (2026-06-10):
  1. Atom model — sources decompose their footprint into finest-grain atoms; one fold serves every read (replaces plan v1's two-mode engine, see §5).
  2. Flat period-total shares for the always-on streams (on-premise provisioned, storage baseline); hourly shares for demand streams. Fixes the hourly-ratio conservation bug.
  3. Service base consumption keeps its explicit term — a service's standing reservation is paid by that service's own jobs (today's semantics).
  4. Both analysis docs were amended in place for the audit findings; this plan never contradicts them.

1. Approach

One concept. Each impact source's footprint decomposes, exactly once, into atoms — the finest (source, stream, containment cell, usage pattern) slices of hourly footprint. Every number the attribution layer ever serves — node totals at any level, links between any two visible columns, skipped columns, source exclusions, conservation checks — is the same operation: group atoms by a key and sum.

Sources own their atom builders (all the physics: binding-resource streams, data→carbon, storage streams, affine power, chassis split). Containers own the dimensionless occurrence / data primitives the builders consume. A single fold in core/attribution/ turns the atom stream into per-level dicts and Sankey links. The whole layer is a pure, lazy, read-time projection over the settled calculated-attribute graph — cached in two tiers (atom lists per (source, phase); folded results per query), both in the flushed render cache, never as model state.

1.1 The atom

An atom is a small frozen value object:

  • value — a labeled hourly kg series (ExplainableHourlyQuantities), e.g. "Server A provisioned footprint via Job J in Step S (Pattern P)";
  • source + stream — who emitted it and which physical stream (provisioned / dynamic / retention / baseline / …) it belongs to;
  • cell coordinates — the source-specific containment fields (job, step, rsn, ef, recn, redn, all optional) plus up, always present. Coarser keys are derived, never stored: uj = up.usage_journey (or edge_usage_journey), country = up.country.

attribution_atoms(phase) yields these one by one — a flat, long-format stream (rows of a table), never a nested dict. Coordinates are named fields, not dict keys; the dict-of-dict-of-dict shape that plan v1 rightly feared simply never exists. The fold accumulates straight into flat {node: series} / {(finer, coarser): series} dicts. Caching is two-tier, matching what depends on what: atom values depend only on (source, phase) — cached at that key (one float32 series ≈ 35 KB per modeled year; a few MB for typical models) — while groupings depend on the query, so folded results are memoized per (phase, visible levels, exclude). One Sankey render builds the atoms once per phase; toggling a skipped column or an exclusion only re-accumulates cached series, rebuilding nothing. Both tiers live in the flushed render cache, never as model state.

Each atom defines its chain — the ordered list of nodes it climbs through, source-ward to System-ward, with absent coordinates simply dropped:

Atom familyCoordinatesChain (≙ Sankey columns)
Server / Storage / ExternalAPI / Network — web (job, step, up) Source → Job → Step → UJ → UP → Country
Server / Storage / ExternalAPI / Network — edge-triggered (job, rsn, ef, up) Source → Job → RSN → EF → EUJ → EUP → Country
Device (step, up) Device → Step → UJ → UP → Country
EdgeDevice (recn, redn, ef, up) EdgeDevice → RECN → REDN → EF → EUJ → EUP → Country

This single structure is what dissolves plan v1's machinery:

  • Neutral vs country-dependent stops being engine machinery. It is just whether CI[up] appears as a factor inside the builder (Network, Device energy, EdgeDevice energy) or not (Server, Storage, Device fabrication). The fold never knows.
  • The edge occurrence ratios become slot enumeration. A need reused across bundles/functions emits one atom per slot — (recn, redn, ef) with the scalar multiplicity o(n, slot, up)/o(n, up) folded into the value. All 21 pairs of edge-analysis.md, including RECN→REDN/REDN→EF with within-journey reuse, are plain group-bys. Same for the (rsn, ef) ratio on the server side.
  • Per-stream relay weights fit naturally. The flat-vs-hourly weight kind (settled decision 2) is private to each builder; no shared weight contract needs to know streams exist.
  • Joint flows need no special resolvers. Job↔Step, or Step↔UP with columns skipped, is a group-by on two keys of the same atoms.

1.2 Per-source atom builders — the only per-source code

Each source class implements one generator, attribution_atoms(phase), documented with the analysis.md formulas. Everything below the table is shared.

Phase and stream are orthogonal, and the phase split survives everywhere. The life-cycle phase (fabrication | energy) is the parameter of every builder and every fold — atoms, node totals and links are always per-phase, so splitting by fabrication vs energy at any column works exactly as today. A stream is the within-phase attribution driver. The provisioned stream shows up in both phases (fabrication rides provisioning entirely; idle energy rides it too, with the same weights):

Cell shares: the two weight kinds, where they live

For a job-leaf source, a cell is one (step, up) or (rsn, ef, up) the job runs in. Two shares exist for every cell, both derived from the same container primitives:

  • c_cell(h) — hourly: cell occurrences(h) / job total occurrences(h), divide_or_fallback(fallback=0). Exact for demand streams: zero occurrences ⇒ zero footprint.
  • C_cell — flat: Σh cell occurrences / Σh job total — a scalar. Used by the always-on streams (on-premise provisioned, storage baseline), which carry footprint at idle hours where c_cell(h) is 0/0 (fallback 0 drops mass, fallback 1 books it once per cell — the bug this plan fixes). Conserves at every hour by construction.

The edge (rsn, ef) slot multiplicity o(rsn, ef, up)/o(rsn, up) is a scalar count ratio on top of either kind.

  • energy_footprint_for_data_volume_and_usage_pattern(data, up) on Network — data→carbon, called by every network atom.

1.3 Container occurrence primitives

Unchanged from the analysis docs — dimensionless, calculated-attribute-style quantities on the containers, consumed by the atom builders (both their hourly values and their period totals):

  • JobBase: get_hourly_avg_occurrences_per_usage_pattern_per_step(up, step) and the per-RSN analogue; compute_hourly_data_transferred_per_usage_pattern_per_step / _per_recurrent_server_need; hourly_data_stored_per_step / _per_recurrent_server_need; the per-step / per-journey / per-RSN / per-edge-journey occurrence regroups.
  • UsageJourneyStep: hourly_avg_occurrences_per_usage_pattern — the Device occupancy primitive (its first calculated attribute), summing over the step's positions in the journey.
  • Edge topology counts (o(n, slot, up), o(rsn, ef, up)) — already materialized or cheap to read off the object lists.

1.4 The fold — core/attribution/

A small module of pure functions, not a ModelingObject. The Sankey feed is one pass:

for atom in atoms(system, phase, exclude): # builders streamed, excluded sources filtered chain = [n for n in atom.chain() if level_of(n) in visible_levels] for node in chain: node_totals[node] += atom.value for a, b in pairwise(chain): links[(a, b)] += atom.value

That loop is the renderer's data layer. Conservation is structural: every atom contributes the same value to each node of its chain and to both links around every middle node, so Σ incoming = node total = Σ outgoing holds at every node by construction — no normalization, no rescaling, anywhere. On top of it, thin memoized reads:

  • node_totals_and_links(system, phase, visible_levels, exclude) — the Sankey feed above.
  • footprint_per_node(level, phase, exclude) / per-source variant — programmatic per-level dicts (replaces attributed_*_footprint_per_source; powers "footprint of this UJ due to Server X" at any level, not just leaves).
  • Skip a column = leave the level out of visible_levels (adjacent columns link directly). Exclude a source = filter its atoms (no rescale). Exclude a subtree = filter atoms by coordinate.

This deletes the entire resolve_attributed_footprint_per_source / _merge_rescaled / generic *_impact_repartition apparatus. Full shape: attribution_sketch.py.

1.5 Caching & invalidation

  • Entire attribution layer = lazy → pay-on-render; zero cost at build and on un-rendered updates.
  • Two cache tiers, matching data dependencies: atom lists per (source, phase) (values are query-invariant — built once, reused by every fold) and folded results per (phase, visible levels, exclude). Skip/exclude interactions re-accumulate, never re-build. Tier 1 is essentially free even for a single render: the series exist in RAM during the fold anyway — caching just retains references (a few MB), and doubles as the debugging/audit view.
  • Cache lifetime ≠ performance story. In the interface context the model is serialized between interactions, so both tiers die with the request — by design the cold render is what must be cheap, and it is: O(#atoms) numpy multiplies + dict accumulations, milliseconds, small next to json_to_system or the plotly assembly. The model graph (calculated attributes) still round-trips through JSON, so a request rebuild recomputes only this thin attribution layer, and only when a Sankey is actually requested. The caches pay off in long-lived processes (notebooks, scripts, tests) and within a request combining a render with per-object reads.
  • Auto-flush every cached property (discover cached_property descriptors via the class MRO, memoized per class). Drops the manual _attributed_footprint_cached_property_names registry and its per-class overrides; fold memos flush the same way.
  • Flush is a flat system-wide sweep over all_linked_objects, once at the end of every ModelingUpdate and after the initial build. No traversal-graph dependency.
  • Invariant: calculated attributes never read attribution results. This one-way rule is what makes wholesale lazy flush correct.

1.6 The renderer — column-walk, geometry only

Sketched in sankey_sketch.py. The renderer asks the fold for (node_totals, links) with its visible levels and excluded sources, then only does presentation: System root + life-cycle-phase columns, object-category / breakdown-by-source decorations (EdgeDevice → EdgeComponent stays the renderer-side hardware axis, unchanged), small-node aggregation, colors, hovers. Deleted: recursive _traverse, resolve_* consumption, flow rescaling. Spacer nodes survive only as geometry — in a mixed web + edge diagram a web Job → Step link visually crosses the edge-only RSN/REDN column; spacers carry it across. They no longer touch values.

3. Cross-cutting concerns

  • Tests: conservation is structural, so the backbone is one generic property test per source: Σ atoms == each stream's footprint == the eager totals, on a model exercising web + edge + reuse + on-premise idle hours. Plus: per-level dicts partition node totals; links-vs-node consistency comes free from the fold but gets a regression test; per-source-exclusion and column-skip equivalence tests; the per-tier helper unit-tested in isolation. Include the long-job repro inherited from the superseded duration-aware-impact-attribution spec: a job whose request_duration outlives its journey window (e.g. a >1h video-generation job in a 3-minute journey) must have its full run-window footprint attributed to its journey/pattern — the request_duration-averaged occurrence primitives make this hold by construction, and the negative-neutral-remainder machinery that used to crash on it is deleted. Existing Sankey/integration totals must be reproduced before old paths are deleted.
  • Serialization: attribution is lazy memoized, not calculated attributes → not serialized — deliberately: persisting render caches would bloat the JSON with hundreds of hourly series to save milliseconds per request (see §1.5 cache-lifetime note). Removing the repartition-weight calculated attributes simplifies the round-trip; JSON schema impact checked at tasks stage (likely a version bump with a no-op upgrade handler dropping stale keys).
  • Docs: architecture.md (attribution boundary, resolved-attribution, cached-property registry sections) and conventions.md (repartition-weight preferences) rewritten to the atom model.

4. Risks

  • Server per-tier provisioned weights — still the single subtlest physics (on-prem vs autoscaling, raw[h] > k−1, flat cell shares). Isolated as a tested helper.
  • Atom enumeration completeness — a missed slot or cell silently drops footprint. Mitigated structurally: the Σ-atoms-equals-source-total test fails loudly on any gap, per stream.
  • Performance on large models — atoms are O(jobs × cells) hourly multiplies at render time. Hundreds of series for typical models (fine); atom lists cached per (source, phase) and folds per query, so values are built once per render and interactions only re-accumulate. If a pathological model hurts, vectorize inside builders — the API doesn't change.
  • Auto-flush global invariant — commits the codebase to "every @cached_property is a flushable read-time projection". True today; must be stated as a rule so it stays true.
  • Renderer rewrite — must reproduce current totals before old paths are deleted (validated by conservation tests, not by keeping a façade).

5. Alternatives considered

  • Plan v1: two-mode engine + source contract (leaf-split vs footprint_at, weighted_redistribution / group_cells_by_ancestor / two joint resolvers / is_under) — superseded: four parallel code paths to audit; the country-dep climb assumed cell→node is a lookup, so the edge occurrence ratios (RECN→REDN/EF reuse, RSN→EF) had no path through it; neutral_joint hard-coded Job/Step occurrence atoms inside the "generic" engine; and the flat-vs-hourly per-stream weights (decision 2) didn't fit a single weight_for(level) contract. The atom model serves every one of those cases with one mechanism.
  • Explicit per-pair methods, no shared fold — every analysis bullet its own method (~60 methods). Most greppable, but massive boilerplate, copy-paste drift risk, and conservation must be re-proven per method instead of structurally.
  • Nested dicts ({job: {cell: {up: series}}}, cached) instead of the flat atom stream — informationally identical (the atoms are the leaves; equivalently a flat dict keyed by the full coordinate tuple), but rejected as the API shape: (a) a nesting order privileges one query axis while our reads group along every axis — each consumer would flatten the tree before aggregating anyway; (b) there is no single nesting scheme across source families (four cell shapes → four traversal paths, the v1 disease); (c) the structure becomes the contract — adding the EF level would have broken every traversal, vs one optional field + one chain() entry with atoms. Caching is orthogonal: the heavy cost (numpy leaf-building) is identical in both shapes, and the atom lists are cached per (source, phase) (§1.5) — that is exactly "cache the nested dict", minus the nesting.
  • Materialize atoms as stored model state — rejected, unchanged from v1: state bloat and explainability noise. Streaming + memoized folds keeps atoms transient while giving the same auditability.
  • Keep generic repartition in ModelingObject — rejected: the framework-layer leak and the root cause of column-skip pain (original sin #2).
  • Eager calculated attributes for fine breakdowns — rejected: only consumed at render; lazy keeps them pay-on-read.
  • Per-edge incremental cache invalidation — rejected: fragile; wholesale flush is correct and simple.

6. Constitutional notes

  • §1.1 three-layer separation — net positive: attribution moves out of abstract_modeling_classes/ into core/, relieving the documented back-edge. The fold imports only upward.
  • §1.2 no backward compatibility — old paths deleted, not shimmed.
  • §1.3 leanness — net deletion of generic machinery; per-source code is one builder each; the fold is ~a screenful.
  • §1.4 doc-as-code — each atom builder and container primitive carries an update_*-style docstring as SSOT, mirroring its analysis.md bullet.
  • §2.3 round-trip — preserved (see §3). No §4 rules touched → no amendment required.
