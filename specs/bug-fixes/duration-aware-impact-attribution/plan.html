Duration-aware impact attribution — Implementation plan

Status: Plan — under review · 2026-06-02 · Spec: spec.html

In one line: impact-repartition weights belong only where a single physical resource is shared among consumers (the source edges). Above that boundary the graph is logical containment, so footprints must be aggregated and routed by occurrence per (source, usage_pattern), not multiplied by normalised shares. Doing so removes the truncation entirely and makes conservation hold by construction.

1. Approach

A source’s footprint is duration-spread correctly, but it is then rolled up to its journey through a chain of normalised shares whose hourly support is the journey’s short activity window. Multiplying a multi-hour footprint by a one-hour share clips the tail at every hop — that is the crash (negative neutral remainder) and the silent under-counting.

The fix is to apply the weight/share mechanism only at the apportioning boundary (one shared resource → its consumers) and replace it with summation + occurrence routing everywhere above. Because occurrences are already tracked per usage pattern (hourly_avg_occurrences_per_usage_pattern), each source’s contribution is carried per (source, usage_pattern) all the way up, and per-pattern carbon provenance is preserved by construction.

EdgeMechanism
source → consumer (Server/Network/Storage/ExternalApi → job; Device → step) weight → normalised share, keyed per (consumer, usage_pattern)
job → step → journey (logical containment) aggregate per (source, pattern); route by occurrence/provenance
journey → pattern direct per-pattern aggregation — no lumped total, no re-split

Per-pattern keying is the usage mechanism (carbon depends on country). Fabrication is country-independent, so it aggregates per-source and occurrence-routed without per-pattern tracking — see §3.

2. What stays, what changes

Stays — source apportioning

Impact sources keep their weight builders and the base normalise-and-multiply path that produces attributed_*_footprint_per_source for their direct consumers. One usage-phase change at the source level: the neutral shared sources (Server/Storage/ExternalApi) currently lump across patterns (job_repartition_weights uses hourly_avg_occurrences_across_usage_patterns); for usage they must key per (job, usage_pattern). Network/Device are already per-country objects and are unaffected. Fabrication keeps per-job keying (§3) — so ServerBase can no longer share one job_repartition_weights dict across both phases; the usage weight becomes per-pattern while the fabrication weight stays per-job (exact factoring left to tasks).

3. The journey → pattern split (the subtle core)

Two facts make this the crux:

  • Tail. The current split multiplies a lumped neutral total by usage_activity_weight (journey concurrency, zero past the journey window). Once the journey total carries the job’s tail, that share drops the tail again at hour 12 — conservation broken even though the crash is gone.
  • Provenance. A footprint’s carbon is per-pattern (network impact ∝ country grid intensity), and a single Job/UsageJourneyStep can be shared across journeys in different countries (verified — each list membership wraps the object in its own ContextualModelingObjectAttribute, so the same object legitimately has multiple journey containers, and nb_of_occurrences_per_container counts them). Routing a lumped total by occurrence then leaks high-carbon network from one pattern into another, and neutral = total − country_dependent goes wrong.

Both dissolve under per-(source, usage_pattern) aggregation for usage: the journey’s per-pattern footprint is a direct sum of per-pattern source contributions — no lumped total, no activity-share re-split, no clock mismatch, provenance preserved. The country-dependent/neutral subtraction machinery is deleted, not fixed.

Fabrication is simpler — it is not carbon-intensity dependent. Embodied emissions are fixed by where hardware is manufactured, not the usage country’s grid, so the provenance argument does not apply and fabrication does not carry per-(source, pattern) information. It still has a run-window tail (server fabrication rides hourly_avg_occurrences), so it needs the same aggregate-and-route fix to stop the logical hops truncating it — but its per-pattern breakdown is a plain occurrence/activity split (distinct device objects keep their own value; the shared neutral splits proportionally), with no carbon leak to guard against. Concretely: fabrication aggregation is per-source and occurrence-routed, kept duration-aware so the tail survives; no journey→pattern re-split machinery.

4. Cross-cutting concerns

  • Conservation is structural, not snapshotted. Aggregation by summation cannot create or destroy emissions, so system and category totals are conserved by construction — pinned by the consistency invariant below, the repro total (test a) and the single-pattern no-regression test (test d), plus the existing suite’s totals. No battery-wide snapshot-equality guard: per-source dicts and per-pattern splits change by design (tails carried, neutral split re-based), so asserting them unchanged would fight the fix.
  • Tests. (a) repro: build the impact-repartition diagram, no raise, journey footprint non-zero in the tail hour and matching the network tail; (b) a jobless device-only step still attributes its device footprint up; (c) a >1h job spread across its run window; (d) all-short-jobs / single-pattern no-regression; (e) a job and a step shared across two journeys in different countries — each country’s intensity stays with its own network/device, neutral server splits by per-pattern occurrence (currently uncovered); (f) edge mirrors of each.
  • Consistency invariant. A child’s routed contributions summed over its parents reproduce its own total, and per-pattern job activity reconciles with hourly_avg_occurrences_per_usage_pattern — a cheap assertion that pins the aggregation.
  • Migration. All affected attributes are calculated (skipped by to_json(save_calculated_attributes=False)); no input/persisted shape changes → no schema migration. Confirm with a repro round-trip.
  • Docs. Rewrite the “Usage attribution boundary” section of architecture.md around the apportioning-vs-aggregation principle; one-line conventions.md note; update affected docstrings (SSOT, §1.4).

5. Risks

  • Per-source dict shape drift (highest). Sankey + skip/exclude resolution depend on the dict structure and intermediate keys (not the values, which change by design). Aggregation must reproduce the same {source_object: footprint} shape. Mitigation: the existing Sankey / resolve_attributed_footprint_per_source tests plus the repro diagram test (a).
  • Blast radius. ~7 container types + source-level keying + edge mirrors. Mitigation: land one level at a time (demand chain, then edge), each checked against the consistency invariant.
  • Latent pre-existing bug. The shared-job-across-countries path is untested and may already be wrong. Mitigation: add test (e) early; if it exposes a discrepancy, surface it — don’t encode old numbers as “expected” (§3.1).
  • Cache invalidation. Removing weight attributes changes the dependency graph; _attributed_footprint_cached_property_names / invalidate_impact_repartition_cache must still flush correctly. Mitigation: existing leaf-mutation test + one for the aggregated path.
  • Sibling-fix ordering. The EcoLogits-server duration-spread fix (../ecologits-server-duration-spread/task.md) produces the spread source footprints this work aggregates — a practical prerequisite.

6. Alternatives considered

  • Duration-aware weights (rebuild shares on the duration clock). Rejected: keeps the weight abstraction where it doesn’t belong and still re-truncates at the device clock and the journey→pattern split. Aggregation removes the machinery instead (§1.3).
  • Lumped neutral total, split by a duration-aware basis. Smaller diff, leaves the source level untouched — but only correct if no step/job is shared across distinct journeys, which is false (§3). It mis-attributes carbon and can re-introduce the crash.
  • Defensive clamp on the negative remainder. Rejected as the fix (§3.1); at most a backstop assertion.

7. Constitutional notes

  • §1.1. Favourable — removes a generic method from the framework layer and moves aggregation into core/; no new upward import, no domain names added to the base.
  • §1.3 / §3.1. Net deletion of machinery, reusing existing per-pattern occurrence quantities; root-cause fix, not a clamp or side-refactor.
  • §2 gates. pytest + mkdocs build --strict + JSON round-trip + CHANGELOG.md. No new class (§2.5 n/a); no amendment required.

8. Settled decisions

  • Mechanism: per-(source, usage_pattern) aggregation for usage; source-level per-pattern keying in scope. (Forced by cross-journey, cross-country step/job sharing.)
  • Fabrication: phase-specific and simpler — per-source, occurrence-routed, duration-aware; no per-pattern carbon tracking (not carbon-intensity dependent).
  • Negative-remainder check, _neutral_activity_share, and the country-dependent/neutral re-split: deleted entirely (dead once aggregation is in).
