Duration-aware impact attribution

Status: Spec — under review · 2026-06-02 · bug-fix (approach a)

1. Problem and audience

One-line: a usage journey’s footprint can never extend past the hour its journey starts — even when the journey fires a backend job that keeps running for hours. Emissions that physically occur later are either mislocated in time or, when one source disagrees, dropped entirely — and that disagreement crashes the Results panel.

Who hits it

  • Modelers of asynchronous / long-running backend work — most visibly AI generation (video, long LLM calls). A user spends a few minutes kicking off a request that then runs for tens of minutes to over an hour. The user journey is short; the job it triggers is long.
  • This is not exotic: e-footprint deliberately separates user time (how long the person interacts) from request duration (how long the backend works), so jobs routinely outlive the journeys that launch them.

Severity

  • Hard failure: opening Results on such a model throws and the panel never renders (the impact-repartition / Sankey computation aborts).
  • Silent inaccuracy even when it doesn’t crash: a long job’s emissions are booked into a single start hour, distorting the hourly profile (which matters for time-varying grid intensity and peak reasoning), and the per-journey/per-pattern split can quietly under-count the part that spills past the journey window.

2. The bug, visualised

The whole problem comes from two different clocks in the model that disagree once a job is longer than the journey that triggered it.

Different parts of the model honour different clocks. Network data transfer (and a normal server’s load) follow the duration clock and correctly spread into 12:00. The attribution that rolls a job’s footprint up to its journey follows the occurrence clock and stops at 11:00. So at 12:00 one side has a value and the other has nothing.

Why the tail gets dropped: it’s truncated at every hop up the tree

A source’s footprint is rolled up to the journey through a chain of nodes, each multiplying by a per-hour share. Those shares are sized from the journey’s short activity window, so they only exist at 11:00. Multiplying a 2-hour footprint by a 1-hour share keeps only the overlap — the 12:00 value is discarded — and this happens again at the next hop, so even fixing one level doesn’t help.

What it should look like after the fix

3. Success criteria

Each is observable on the repro model and/or as a test.

  • Opening Results on a model with a job longer than its journey renders the chart and the impact-repartition diagram — no error.
  • A journey’s attributed footprint spans the run window of its jobs: emissions appear in every hour the work actually happens, not only the journey’s start hour.
  • The relationship “a journey’s total footprint is at least its country-dependent (device + network) part, every hour” holds because the numbers are consistent, not because a negative remainder was clamped away.
  • Conservation: for any model, the system’s total footprint and each top-level category total are unchanged to within float noise — this fix moves emissions in time and across the attribution tree, it does not create or destroy any.
  • No regression for models whose jobs are all shorter than an hour: results are identical to today.
  • The same guarantees hold for the edge (deployment-driven) paradigm, which has the same attribution structure.

4. Scope

Capability-level only. The concrete classes, methods, weight formulas, and the ordered refactor live in plan.html (to be written in the spec-plan stage). A starter set of technical anchors from the investigation is parked in §7 so the plan stage doesn’t begin cold.
In scope
  • Making attribution of a source’s footprint to journeys / usage patterns honour the full hourly window over which that footprint exists, so emissions are never silently truncated when work outlives its journey.
  • Resolving the disagreement between the two clocks so the every-hour consistency relationship holds without clamping.
  • Covering both the demand-driven (web) and deployment-driven (edge) paradigms.
  • The decision, already taken with the maintainer, that a journey owns its job for the job’s full lifetime — its footprint legitimately extends into hours when the user is no longer active (“approach a”, see §6).
Out of scope (this iteration)
  • Spreading the EcoLogits external-API server’s own per-request impact over the request duration. That is a real, related inaccuracy but is handled as a separate, smaller fix — see the sibling task ../ecologits-server-duration-spread/task.md. (It is a prerequisite in practice: it produces the spread footprints this fix then carries through.)
  • Re-deciding how request duration itself is derived (e.g. whether ~62 min generation latency for a 4-minute video is realistic) — that is an upstream data/assumption question.
  • The defensive clamp on the negative remainder. It may be kept as a backstop, but it is explicitly not the fix (constitution §3.1: never paper over a bug).

5. Constraints

  • Conservation of totals is the dominant correctness constraint: any reshaping of attribution must preserve per-model totals. Guard with golden snapshots over a battery of existing models before changing anything.
  • Constitution §3.1 — fix the root cause; the clamp is a backstop at most, not the resolution.
  • Constitution §1.3 (leanness) — reuse the existing duration-spread idiom already used by network/storage rather than inventing a parallel mechanism.
  • Explainability — every reshaped attribute must keep a sensible label and calculation graph; the interface and mkdocs read these.
  • JSON round-trip preserved; a schema migration is added only if any persisted attribute changes shape.
  • Performance — shares become slightly longer hourly series across many objects; keep an eye on recompute cost on large models.

6. Open questions

  • Resolved — the semantic fork. Two readings were on the table: (a) a journey owns its job’s footprint for the job’s full lifetime, so the journey’s footprint extends into hours the user is no longer active; or (b) the orphan tail stays on the source (server/network) and is simply not attributed to any journey in those hours. Decision: approach (a). This spec describes (a).
  • What is the right per-hour share in an hour where several long jobs from different journeys overlap but the journeys themselves are no longer active? (Normalisation basis for the duration-aware shares — a plan-stage question.)
  • Should the every-hour consistency check become a redundant assertion (always true once shares are duration-aware) or be removed? Decide once the mechanism is in.

Technical anchors from the investigation (plan-stage material — verify before relying on)
  • Where it throws: the per-usage-pattern energy attribution on the usage journey, which computes neutral = attributed_energy_footprint − Σ country_dependent_usage_footprint and raises when it goes negative beyond tolerance. An identical construct exists on the edge usage journey — fix both.
  • The truncating mechanism: the generic _compute_default_impact_repartition_weight shapes a child’s share from its container’s weight series times a scalar occurrence count, so the share inherits the container (journey) time-window, never the child (job) window. The normalised share then truncates any longer footprint on multiplication. Overriding only the job level was tested and is insufficient — the tail dies again at step→journey and journey→pattern.
  • Repro facts: journey duration ≈ 3 min; the “49 x Video generation on Kling” job has request duration ≈ 62 min (it is a single ~4-min video, i.e. 49× the 5 s base clip — not 49 separate calls); its network footprint is [2.35e-4, 4.9e-6] kg over two hours while the journey’s attributed footprint is [3.86, 0]; the shortfall (4.9e-6) just exceeds the relative tolerance (3.86e-6).
  • Conservation check value: system.total_footprint.sum() ≈ 4.2747 kg for the repro model — must be unchanged after the fix.
  • Reference pattern to mirror for duration-aware spreading: the network/storage per-hour exchange (value × 1h/request_duration × hourly_avg_occurrences).
  • Surface to audit: every update_*_impact_repartition_weights builder (generic default; external-API server; device; storage; country; usage journey / usage pattern; and the edge mirrors), the impact-repartition cache invalidation, and the Sankey resolver that consumes the per-source attribution dicts.
